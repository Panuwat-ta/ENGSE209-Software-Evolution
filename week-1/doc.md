# เอกสารความเข้าใจระบบเบื้องต้น (System Understanding Report)
**วิชา: Software Evolution and Maintenance (Week 1)**

เอกสารฉบับนี้จัดทำขึ้นเพื่อสรุปความเข้าใจและการวิเคราะห์ระบบจัดการคลังสินค้าเบื้องต้น (Inventory System) โดยครอบคลุมถึงฟังก์ชันการทำงานปัจจุบัน เทคโนโลยีที่ใช้งาน และการวิเคราะห์จุดที่เป็นช่องโหว่/จุดเสี่ยงภายในซอร์สโค้ด

---

## 1. ฟังก์ชันการทำงานของระบบในปัจจุบัน (System Functionality)
ระบบปัจจุบันเป็นแอปพลิเคชันประเภท Command-Line Interface (CLI) สำหรับจัดการคลังสินค้า โดยมีเมนูหลักทั้งหมด 5 เมนู ดังนี้:

| เมนูที่ | ชื่อเมนู | รายละเอียดการทำงาน |
| :---: | :--- | :--- |
| **1** | **Show all** | แสดงรายการสินค้าทั้งหมดที่มีอยู่ในคลังสินค้า โดยดึงข้อมูลมาจากไฟล์ฐานข้อมูลและจัดรูปแบบการแสดงผลเป็น: <br>`ID: [รหัส] | Name: [ชื่อ] | Stock: [จำนวนคงเหลือ] | Price: [ราคา] THB | Type: [ประเภทสินค้า]` |
| **2** | **Add or Update** | เพิ่มสินค้าใหม่หรืออัปเดตสินค้าเดิม โดยรับค่า ID, Name, Quantity, Price, และ Category จากผู้ใช้ จากนั้นจะเขียนทับหรือสร้างข้อมูลชุดนั้นลงในคลังสินค้าและบันทึกลงไฟล์ทันที |
| **3** | **Out** | หักจำนวนสินค้าออกจากสต็อก (ตัดยอดจำหน่าย) โดยระบุ ID และจำนวนที่ต้องการตัด หากสต็อกเพียงพอ ระบบจะคำนวณยอดใหม่และบันทึกข้อมูล พร้อมมีระบบแจ้งเตือนแบบเร่งด่วนหากจำนวนสินค้าในสต็อกเหลือน้อยกว่า 5 ชิ้น (`!!! WARNING: ITEM IS RUNNING VERY LOW IN STOCK !!!`) |
| **4** | **Check Check** | สรุปรายงานคลังสินค้าเบื้องต้น ได้แก่:<br>1. จำนวนประเภทสินค้าทั้งหมด (Total product types)<br>2. มูลค่ารวมของสินค้าทั้งหมดในคลัง (Total inventory value)<br>3. รายชื่อสินค้าที่มีสต็อกต่ำกว่า 10 ชิ้น เพื่อเตือนให้เตรียมสั่งซื้อเพิ่ม (Alert low stock) |
| **5** | **Exit** | ออกจากโปรแกรม |

## 2. เทคโนโลยีที่ใช้งาน (Technology Stack)
*   **ภาษาคอมพิวเตอร์ (Language):** Python (มาตรฐานสากล ใช้ไลบรารีพื้นฐานของระบบ)
*   **เฟรมเวิร์ก (Framework):** ไม่มีการใช้ Framework (เป็นระบบที่พัฒนาขึ้นโดยใช้ Standard Python Library)
*   **ระบบฐานข้อมูล (Database):** ข้อมูลถูกจัดเก็บในรูปแบบ **JSON File** ผ่านไฟล์ [data.json] โดยเมื่อเปิดระบบจะดึงข้อมูลมาเก็บไว้ในโครงสร้างข้อมูลแบบ Python Dictionary (ในหน่วยความจำชั่วคราว) และเซฟข้อมูลกลับไปที่ไฟล์ JSON เมื่อมีการแก้ไขข้อมูลสินค้า

## 3. จุดที่มีความคลุมเครือ (งง) หรือเป็นจุดเสี่ยงที่สุดในโค้ด (Code Issues & Risks)

จากการทบทวนโค้ด (Code Review) พบจุดเสี่ยงและจุดที่น่าสงสัยดังนี้:

### จุดเสี่ยงขั้นรุนแรง (Critical Risks)

#### 1. การขาดระบบดักจับข้อผิดพลาดของข้อมูลนำเข้า (No Input Validation)
*   **จุดเกิดเหตุ:** บรรทัดที่ 46-47 และ 63 ในไฟล์
    ```python
    c = int(input("Enter Qty: "))
    d = float(input("Enter Price: "))
    # และ
    amt = int(input("How many items out?: "))
    ```
*   **ความเสี่ยง:** หากผู้ป้อนข้อมูลกรอกข้อมูลที่ไม่ใช่ตัวเลข (เช่น ใส่ตัวอักษร "abc" หรือปล่อยว่าง) โปรแกรมจะเกิดข้อผิดพลาด `ValueError` และ**พังทันที (Crash)** ทำให้ผู้ใช้ต้องเปิดโปรแกรมใหม่และอาจสูญเสียข้อมูลที่ยังไม่ได้เซฟได้อย่างง่ายดาย

#### 2. ช่องโหว่จากการใส่ค่าติดลบ (Negative Value Vulnerability)
*   **จุดเกิดเหตุ:** เมนู 2 (Add or Update) และ เมนู 3 (Out)
*   **ความเสี่ยง:**
    *   ผู้ใช้สามารถกรอกจำนวนสินค้าหรือราคาติดลบในขั้นตอนการเพิ่มสินค้าได้ (เช่น จำนวน -10 ชิ้น หรือ ราคา -100 บาท) ซึ่งไม่มีทางเกิดขึ้นจริงในทางปฏิบัติ
    *   ในขั้นตอนการตัดสต็อก (เมนู Out) หากผู้ใช้ป้อนจำนวนสินค้าที่จะเอาออกเป็นค่าติดลบ (เช่น `-50`) โค้ดจะทำงาน: `x[id_to_cut]['q'] = x[id_to_cut]['q'] - (-50)` ซึ่งเป็นการ**เพิ่มจำนวนสินค้าในสต็อกแทนการตัดออก** ส่งผลให้ระบบทำงานผิดพลาดอย่างร้ายแรง

#### 3. ความเสี่ยงข้อมูลสูญหาย/เสียหายเมื่อเกิดการขัดข้อง (Data Corruption/Loss Risk)
*   **จุดเกิดเหตุ:** ฟังก์ชัน `save()` บรรทัดที่ 21-23
    ```python
    def save():
        with open(db, 'w') as f:
            json.dump(x, f)
    ```
*   **ความเสี่ยง:** การบันทึกข้อมูลใช้วิธีเปิดไฟล์ด้วยโหมด `'w'` ซึ่งจะล้างข้อมูลเดิมในไฟล์ JSON ออกทั้งหมดทันทีก่อนเขียนใหม่ หากเครื่องคอมพิวเตอร์เกิดไฟดับ หรือโปรแกรมขัดข้องระหว่างที่ฟังก์ชันนี้กำลังเขียนไฟล์ ข้อมูลในคลังสินค้าทั้งหมดจะเสียหายและสูญหายอย่างถาวร (ไม่มีการใช้ Temporary file หรือ Backup ก่อนเขียนไฟล์)

### จุดชวนงงและประสิทธิภาพของโค้ด (Smells & Design Issues)

#### 1. การตั้งชื่อตัวแปรที่ไม่มีความหมาย (Poor Naming Conventions)
*   **จุดเกิดเหตุ:** ตัวแปรระดับ Global และ Local ทั่วทั้งไฟล์
    *   `x` คือ ตัวแปรเก็บข้อมูลสินค้าทั้งหมดในคลัง (ควรเปลี่ยนเป็น `inventory` หรือ `products`)
    *   `db` คือ ชื่อไฟล์ JSON (ควรเปลี่ยนเป็น `DATABASE_FILE`)
    *   `a`, `b`, `c`, `d`, `e` ในขั้นตอนรับข้อมูลเมนู 2 (ควรเปลี่ยนเป็น `prod_id`, `prod_name`, `quantity`, `price`, `category`)
*   **ความงง:** การใช้ตัวแปรอักษรเดี่ยวทำให้โค้ดอ่านยากมากเมื่อระบบขยายตัวใหญ่ขึ้น และไม่เป็นไปตามมาตรฐาน PEP 8 ของ Python

#### 2. บล็อกเงื่อนไขที่ซ้ำซ้อนอย่างไม่จำเป็น (Redundant Code)
*   **จุดเกิดเหตุ:** บรรทัดที่ 51-55
    ```python
    # This logic updates or creates
    if a in x:
        # if already have, just add qty? or overwrite? Let's overwrite!
        x[a] = {"n": b, "q": c, "p": d, "c": e}
    else:
        x[a] = {"n": b, "q": c, "p": d, "c": e}
    ```
*   **ความงง:** คำสั่งในบล็อก `if` และ `else` เหมือนกันทุกประการ ไม่ว่าจะรหัสสินค้ามีอยู่แล้วหรือเป็นสินค้าใหม่ก็เขียนทับด้วยโครงสร้างข้อมูลเดียวกัน การเขียนบล็อกเงื่อนไขนี้จึงไม่มีประโยชน์และทำให้โค้ดสับสน ควรยุบรวมเหลือเพียงบรรทัดเดียว คือ `x[a] = {"n": b, "q": c, "p": d, "c": e}`

#### 3. การใช้งานตัวแปร Global (Global State Mutation)
*   **จุดเกิดเหตุ:** การกำหนดและใช้ `global x` ในฟังก์ชัน `load()`
*   **ความเสี่ยง:** การเปลี่ยนแปลงสถานะตัวแปรหลักผ่าน `global` ทำให้ตรวจสอบยากว่าข้อมูลเปลี่ยนไปในขั้นตอนใดบ้าง และเป็นอุปสรรคอย่างมากต่อการเขียน Unit Test เพื่อทดสอบระบบในอนาคต
