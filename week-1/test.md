## 1. บทบาทและหน้าที่ภายในทีมพัฒนา (DevTeam Roles)

เพื่อประสิทธิภาพในการบำรุงรักษาและปรับปรุงระบบ สมาชิกในทีมได้ตกลงแบ่งบทบาทความรับผิดชอบหลักดังนี้:

| บทบาท (Role) | หน้าที่รับผิดชอบ (Key Responsibilities) |
| :--- | :--- |
| **Project Manager (PM)** | * บริหารจัดการภาพรวมโครงการ กำหนดกรอบเวลา (Timeline) และงานในแต่ละช่วง<br>* ติดตามความคืบหน้าของทีม ประสานงาน และจัดการอุปสรรคต่าง ๆ ของทีมพัฒนา |
| **Tech Lead / Architect** | * ออกแบบแนวทางการปรับโครงสร้างโค้ด (Refactoring Architecture)<br>* ตรวจสอบคุณภาพของโค้ด (Code Review) และกำหนดแนวทางเขียนโค้ดที่ถูกต้องตามมาตรฐาน (PEP 8) |
| **Developer** | * ดำเนินการปรับปรุง แก้ไขบั๊ก และพัฒนาระบบตามที่ Tech Lead มอบหมาย<br>* เขียนโค้ดในส่วนของการดักจับข้อผิดพลาด (Input Validation) และการจัดเก็บข้อมูลให้มีความเสถียร |
| **QA / Tester** | * ออกแบบกรณีการทดสอบ (Test Cases) ครอบคลุมการทำงานทั่วไปและเคสขอบเขตที่เป็นอันตราย (Edge Cases)<br>* จัดทำเอกสารชุดทดสอบและทำ Automated Unit Testing เพื่อยืนยันคุณภาพของระบบ |

## 2. เป้าหมายโครงการ (Project Objectives)

1. **ปรับปรุงเสถียรภาพการทำงาน (Stability Improvement):** ทำให้ระบบจัดการสินค้ามีความทนทานต่อข้อผิดพลาด (Fault Tolerance) ไม่หยุดทำงานหรือพังลงจากการรับข้อมูลนำเข้าที่ไม่ถูกต้อง (Crash Prevention)
2. **ปรับปรุงโครงสร้างโค้ด (Refactoring):** ทำความสะอาดโค้ดเดิม ยุบโค้ดส่วนที่ซ้ำซ้อน แยกฟังก์ชันออกเป็น模组 (Modularized) ให้ชัดเจน และกำจัดการใช้ Global Mutable State เพื่อให้โค้ดอ่านง่ายและง่ายต่อการบำรุงรักษาในระยะยาว
3. **รักษาความสมบูรณ์ของข้อมูล (Data Integrity):** ปรับปรุงฟังก์ชันการบันทึกฐานข้อมูล JSON ให้ปลอดภัย ป้องกันความเสี่ยงจากการที่ไฟล์ข้อมูลเสียหรือสูญหายเมื่อการทำงานหยุดชะงักลงกะทันหัน
4. **สร้างระบบทดสอบแบบอัตโนมัติ (Automated Testing):** จัดทำชุดทดสอบเพื่อให้มั่นใจว่าการปรับปรุงโค้ดเดิมจะไม่ส่งผลเสียต่อการทำงานที่มีอยู่ (Regression Testing)

## 3. ขอบเขตโครงการ (High-level Scope)

### 📌 สิ่งที่อยู่ภายในขอบเขต (In-Scope)
*   **การจัดการ Input Validation และ Exception Handling:** ดักจับข้อผิดพลาดจากการป้อนข้อมูลในส่วนของการรับราคา จำนวน และคัดกรองการรับค่าประเภทจำนวนติดลบ
*   **การปรับปรุงโครงสร้าง (Refactoring):** 
    *   เปลี่ยนชื่อตัวแปรที่กำกวมทั้งหมด (`x`, `db`, `a`, `b`, `c`, `d`, `e`) ให้มีความหมายตามมาตรฐาน
    *   แยกตรรกะทางธุรกิจ (Business Logic) ออกจากส่วนแสดงผลทางหน้าจอ (CLI Presentation Layer)
    *   ลบบล็อกคำสั่งเงื่อนไขที่ซ้ำซ้อนในฟังก์ชัน Add/Update
*   **การเพิ่มระดับความปลอดภัยในการบันทึกข้อมูล:** ปรับปรุงส่วนการเซฟไฟล์ JSON โดยใช้หลักการ Atomic Write (บันทึกไฟล์ชั่วคราวก่อนทำการย้ายทับ) เพื่อป้องกันข้อมูลสูญหาย
*   **การเขียนชุดทดสอบอัตโนมัติ:** สร้าง Unit Tests เพื่อทดสอบฟังก์ชันทางธุรกิจ เช่น การคำนวณมูลค่ารวมคลังสินค้า และการคำนวณจำนวนสต็อกหลังจากการจ่ายออก

### 🚫 สิ่งที่อยู่นอกขอบเขตอย่างเด็ดขาด (Out-of-Scope)
*   **การเปลี่ยนรูปแบบ User Interface:** จะไม่มีการพัฒนาหน้าต่าง GUI (Graphical User Interface) หรือเปลี่ยนระบบให้ทำงานเป็น Web Application ระบบยังคงรักษาขอบเขตเป็นโปรแกรมบน Terminal (CLI)
*   **การเปลี่ยนเทคโนโลยีฐานข้อมูลหลัก:** จะไม่มีการเชื่อมต่อกับระบบฐานข้อมูล SQL หรือ NoSQL ภายนอก (เช่น MySQL, MongoDB) แต่ยังรักษารูปแบบฐานข้อมูล JSON ไฟล์เดียวแบบดั้งเดิม
*   **การเพิ่มโมดูลฟังก์ชันใหม่นอกเหนือจากสต็อกสินค้า:** จะไม่มีการเขียนฟีเจอร์ที่ไม่เกี่ยวข้องกับคลังสินค้า เช่น ระบบชำระเงินออนไลน์ ระบบสมาชิก (Membership) หรือระบบทำโปรโมชั่นสินค้า

## 4. ผลลัพธ์ที่ส่งมอบ (Key Deliverables)

1. **Refactored CLI Inventory Code:** ซอร์สโค้ดของระบบจัดการสต็อกสินค้าในภาษา Python ที่ผ่านการปรับปรุงสถาปัตยกรรม การจัดการข้อผิดพลาด และโครงสร้างตัวแปรเรียบร้อยแล้ว
2. **Automated Test Suite:** ไฟล์สคริปต์สำหรับการทำ Unit Tests เพื่อนำไปใช้รันสอบทานความถูกต้องของระบบโดยฝ่าย QA
3. **System Documentation:** เอกสารอธิบายการออกแบบสถาปัตยกรรมระบบใหม่ (System Architecture Document) และวิธีการติดตั้งใช้งานโปรแกรม
4. **Bug-free Validation Log:** รายงานการตรวจสอบความถูกต้องโดย QA เพื่อแสดงให้เห็นว่าระบบเดิมที่มีข้อบกพร่องได้รับการแก้ไขจนเสร็จสิ้น

## 5. ความเสี่ยงเบื้องต้น (Initial Risks)

1. **ความเสี่ยงจากการพังยับเนื่องจากการเปลี่ยนโค้ด (Regression Risk):**
   *   *รายละเอียด:* การนำตัวแปร Global `x` ออกและแปลงระบบเป็นแบบแยกส่วน อาจจะส่งผลให้โค้ดส่วนอื่น ๆ ที่ยังคงเรียกหาตัวแปร global ทำงานผิดปกติโดยไม่ได้ตั้งใจ
   *   *แนวทางจัดการ:* ต้องทำ Refactoring ทีละส่วน และมีขั้นตอนรันชุดทดสอบความถูกต้องอย่างสม่ำเสมอหลังทำการเปลี่ยนแปลง
2. **ความเสี่ยงของโครงสร้างข้อมูลเดิมเสียหาย (Data Schema Incompatibility Risk):**
   *   *รายละเอียด:* ข้อมูลเดิมในไฟล์ [data.json](file:///D:/work/Software%20Evolution%20and%20Maintenance/week-1/data.json) อาจจะมีข้อมูลที่เป็นรูปแบบไม่สมบูรณ์ (เช่น สต็อกหรือราคาถูกบันทึกเป็นข้อความแทนตัวเลข) ซึ่งแอปพลิเคชันเวอร์ชันใหม่อาจพังเมื่อเปิดอ่านข้อมูลเดิม
   *   *แนวทางจัดการ:* เขียนสคริปต์สแกนตรวจสอบข้อมูลเดิม (Data Migration/Sanitizer script) เพื่อล้างหรือแปลงค่าข้อมูลเก่าให้สอดรับกับเวอร์ชันใหม่
3. **ความเสี่ยงจากการไม่มีระบบทดสอบเดิมรองรับ (Lack of Legacy Tests Risk):**
   *   *รายละเอียด:* ซอฟต์แวร์ต้นฉบับไม่มีชุดทดสอบอัตโนมัติมาให้ ทำให้ทีมพัฒนาไม่สามารถทราบได้ทันทีว่าการแก้ไขจุดใดจุดหนึ่งไปขัดแย้งกับตรรกะเดิมของลูกค้าหรือไม่
   *   *แนวทางจัดการ:* บังคับให้ QA เข้ามามีบทบาทและเขียน Test Case ที่ครอบคลุมฟังก์ชันการทำงานปัจจุบันก่อนที่ Developer จะเริ่มแก้ไขโค้ดโครงสร้างหลัก
