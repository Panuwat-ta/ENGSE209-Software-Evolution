**บอร์ดติดตามและจัดการงานอย่างละเอียด (Detailed Dev Board)**

บอร์ดนี้นำเสนอรายละเอียดของงาน (Tasks) ระดับความสำคัญ (Priority) เงื่อนไขในการยอมรับงาน (Definition of Done) และผู้รับผิดชอบงานในแต่ละส่วน เพื่อใช้ควบคุมและติดตามความคืบหน้าของการปรับปรุงซอฟต์แวร์ระบบจัดการคลังสินค้า รวมถึงการเพิ่มระบบสมาชิกสะสมส่วนลด (Member Discount) และการย้ายระบบจัดเก็บข้อมูลจาก JSON ไปยังฐานข้อมูล SQLite เพื่อความปลอดภัยและเสถียรภาพของข้อมูล

    * [ ] นำกระบวนการ SQLite Transaction (Commit / Rollback) มาใช้ควบคุมความถูกต้องในการเขียนสินค้าและสมาชิกพร้อมๆ กัน

#### TASK-12: Implement SQLite Database Layer and Data Migration Script
* ความสำคัญ: สูง (High)
* ผู้รับผิดชอบ: Developer
* รายละเอียด: สร้างและจัดการฐานข้อมูล SQLite (ระบบจะถูกย้ายจาก JSON ไปใช้ SQLite Database เพื่อความปลอดภัยของข้อมูล) พร้อมเขียนสคริปต์ย้ายฐานข้อมูลเพื่อดึงข้อมูลสินค้าจาก data.json เดิมเข้าตารางของ SQLite และป้องกันปัญหา SQL Injection ด้วยการใช้ Parameterized Query
* รายการย่อยที่ต้องทำ (Sub-tasks Checklists):
    * [ ] สร้างโมดูล database.py เพื่อใช้จัดการการเชื่อมต่อ SQLite (Connection)
    * [ ] เขียนสคริปต์ SQL เพื่อสร้างตาราง products และ members
    * [ ] เขียนสคริปต์ migrate.py สำหรับดึงข้อมูลจาก data.json และนำไป Insert ลงใน SQLite products table
    * [ ] ปรับปรุงคลาส InventoryManager ให้เชื่อมโยงกับ SQLite แทนไฟล์ JSON โดยการใช้คำสั่ง SQL ที่ปลอดภัยจาก SQL Injection (Parameterized Query)
* เงื่อนไขความสำเร็จ (Definition of Done - DoD):
    * ข้อมูลจาก data.json ย้ายเข้าไปยัง SQLite สำเร็จและระบบสามารถใช้งานอ่านและอัปเดตสต็อกผ่าน SQLite แทน JSON ได้โดยสมบูรณ์
    * รหัสคำสั่ง SQL ทั้งหมดใช้ Parameterized Query 100% ไม่มีช่องโหว่ SQL Injection

## ทะเบียนตารางความเสี่ยง (Risk Register) และแผนบรรเทาความเสี่ยง (Mitigation Plan)

ตารางความเสี่ยงนี้มาจากการประเมินปัญหาที่เกิดขึ้นจากโครงสร้างโค้ดดั้งเดิมที่ทีมงานได้ร่วมกันวิเคราะห์และทบทวนอย่างละเอียด โดยครอบคลุมถึงผลกระทบและแนวทางการบรรเทาความเสี่ยงในการวิวัฒนาการซอฟต์แวร์สู่ระบบใหม่

| ลำดับ | หัวข้อความเสี่ยง (Risk Topic) | ผลกระทบต่อระบบ (Impact) | แผนบรรเทาความเสี่ยง (Mitigation Plan) |
| :---: | :--- | :--- | :--- |
| 1 | **ความเสี่ยงจากการขัดแย้งของสถานะตัวแปรระดับ Global (Global State Conflict Risk)** | การนำระบบสมาชิกและระบบส่วนลดสมาชิกเข้ามาเพิ่มเติม หากยังคงใช้โครงสร้างเดิมที่มีการประกาศ global x ในการอัปเดตข้อมูล อาจทำให้เกิดการเขียนทับข้อมูลโดยไม่ได้ตั้งใจ หรือตัวแปรสินค้าและสมาชิกเข้ามาปะปนกันจนเกิดข้อผิดพลาดในการคำนวณราคาสินค้าหรือส่วนลด และทำ Unit Test ได้ยากมาก | **ทำการแปลงระบบเป็น OOP และแบ่ง Layer ของซอฟต์แวร์ (MVC Pattern)**:<br>ทำการ Refactor โดยจัดกลุ่มโค้ดแยกส่วน Business Logic และ Data Access ออกจาก Presentation Layer (CLI) ในรูปของ Class เช่น InventoryManager และ MemberManager โดยให้สื่อสารข้อมูลผ่าน Parameter หรือ Return Value แทนการเรียกหาตัวแปร global ตรงๆ และมีชุดทดสอบ Unit Test เพื่อป้องกันผลกระทบย้อนหลัง |
| 2 | **ความเสี่ยงจากการเขียนทับไฟล์ข้อมูลดั้งเดิมโดยตรงแล้วแอปพลิเคชันล่ม (Data Corruption during Monolithic JSON Write)** | ฟังก์ชันการเซฟข้อมูลเดิมใช้วิธี json.dump ลงในไฟล์ data.json โดยตรง ซึ่งมีความเสี่ยงสูงมากที่ข้อมูลจะสูญหายทั้งหมดหากเกิดกรณีแอปพลิเคชันค้าง ไฟดับ หรือไม่มีสิทธิ์เข้าถึงดิสก์ในขณะที่กำลังบันทึก (เนื่องจากเนื้อหาไฟล์เก่าจะโดนล้างเป็นค่าว่างทันทีที่คำสั่งเริ่มรัน) ส่งผลเสียต่อข้อมูลสินค้าและข้อมูลสมาชิกใหม่ที่จะนำเข้าสู่ระบบ | **เปลี่ยนผ่านระบบจัดเก็บข้อมูลไปเป็นฐานข้อมูล SQLite**:<br>ย้ายการจัดเก็บข้อมูลจากไฟล์ JSON ที่ไม่มีกลไกควบคุมการบันทึก ไปใช้ฐานข้อมูล SQLite ซึ่งสนับสนุนคุณสมบัติ ACID และมีระบบ Journaling ป้องกันความเสียหายของข้อมูล รวมถึงการนำกระบวนการ SQLite Transaction (Begin, Commit, Rollback) มาใช้เพื่อให้แน่ใจว่า ข้อมูลจะถูกบันทึกลงฐานข้อมูลอย่างสมบูรณ์แบบร้อยเปอร์เซ็นต์ หรือยกเลิกการเขียนเพื่อรักษาความสมบูรณ์ของระบบหากเกิดข้อผิดพลาด |
| 3 | **ความเสี่ยงจากช่องโหว่ SQL Injection และการรับประเภทข้อมูลนำเข้าผิดประเภท (SQL Injection and Input Type Mismatch Risk)** | เมื่อเปลี่ยนการจัดเก็บข้อมูลเป็น SQLite Database หากนักพัฒนาเขียนโปรแกรมติดต่อกับฐานข้อมูลด้วยวิธีการเชื่อมต่อข้อความ (String Concatenation) โดยตรง จะเปิดช่องโหว่ SQL Injection ให้ผู้ใช้งานที่ไม่หวังดีสามารถพิมพ์ข้อมูลควบคุมระบบฐานข้อมูลเพื่อขโมยหรือทำลายข้อมูล และหากยังขาดการทำ Input Validation เมื่อรับค่าราคาและจำนวนสินค้าเป็นตัวอักษร ฐานข้อมูลจะส่งผ่าน Error และทำให้ระบบล่มกระทันหัน | **บังคับใช้ Parameterized Query และระบบ Input Validation อย่างเคร่งครัด**:<br>1. บังคับใช้คำสั่งการสืบค้นข้อมูลในลักษณะ Parameterized Query (เช่น cursor.execute("SELECT * FROM members WHERE member_id = ?", (id,))) เพื่อป้องกันช่องโหว่ SQL Injection<br>2. เขียนฟังก์ชัน Helper สำหรับตรวจสอบข้อมูลนำเข้า (Input Validator) ในเลเยอร์ CLI เพื่อคัดกรองตัวเลขที่ไม่ใช่ลบ และดักจับ ValueError ด้วย try-except ก่อนส่งค่าเข้าสู่ระบบประมวลผลฐานข้อมูล SQLite |
| 4 | **ความเสี่ยงจากการเข้ารหัสภาษาไทยผิดเพี้ยนในการจัดเก็บข้อมูล (Encoding Character Mismatch Risk)** | ในโค้ดดั้งเดิมไม่ได้กำหนดค่า encoding='utf-8' ในส่วนเปิดไฟล์ข้อมูล ทำให้เมื่อรันโปรแกรมบนระบบปฏิบัติการอื่น เช่น Windows ที่ใช้ Locale เป็น CP1252 ชื่อสินค้าหรือสมาชิกที่เป็นภาษาไทยจะเพี้ยนกลายเป็นอักษรต่างดาว และเมื่อย้ายไป SQLite อาจเกิดการขัดข้องในการเปรียบเทียบข้อมูล String หรือการบันทึกภาษาไทย | **ตั้งค่า Encoding ให้เป็นมาตรฐาน UTF-8 เสมอ**:<br>กำหนดให้ตารางในฐานข้อมูล SQLite และการสืบค้นคำสั่งระบุประเภทการรับส่งข้อมูลผ่าน UTF-8 เสนอแนวทางปฏิบัติตั้งแต่ระดับ Connection และสร้างชุดทดสอบ Integration Test ที่ดึงชื่อและข้อมูลที่เป็นภาษาไทยและอักขระพิเศษเพื่อยืนยันการจัดเก็บและดึงข้อมูลว่าทำงานได้ถูกต้องครบถ้วน |
